Acceptance rate is a bigger problem than I realized. I see riders at 30–50% acceptance rates routinely. That's a massive hidden lever — a rider doing 18 rides with 35% acceptance could potentially do 25+ with 70% acceptance. The current design only tracks rides; it ignores this.


How to handle the timing problem (the bigger question)
The fix is simple in concept: decouple the target pipeline from the approval pipeline. They serve different purposes and don't need the same speed.
Pipeline A — Target System (Fast, Automated, No Humans)

04:00 AM: Yango sync auto-runs, pulls yesterday's raw data
04:30 AM: System recalculates today's target using raw Yango data
05:30 AM: Targets ready, visible on Guard's screen
06:00 AM: Riders leave with target based on yesterday's actual performance

Pipeline B — Accounting Pipeline (Slow, Verified, Humans Required)

09:00–11:00: Daily Log Operator verifies synced data
11:00–14:00: Accountant approves logs, approves cash collection
This pipeline drives salary, variance investigation, P&L — not targets

Why this works:

Yango IS the source of truth for ride counts. Once Yango has it, we can trust it for targets.
A slightly wrong target (because data hasn't been human-verified) costs almost nothing — the rider has a slightly easier or harder day.
A slow target costs more — the whole feedback loop breaks.
Human verification still happens, just for the things that need it (cash, salary).

Edge cases handled:

Yango sync fails at 04:00: Use last successful day's data + send alert to Daily Log Operator. Guard sees target as normal; rider doesn't notice.
Rider has 0 rides yesterday in Yango data: Treat as absent, exclude from 7-day average. If later turns out to be gaming (rider stayed offline), discipline through normal channels — doesn't affect target system.
Yango sync runs but data looks anomalous: Target gets set anyway. Daily Log Operator's later verification flags it for the day, but next day's target self-corrects.

This is one schema change to EFMS — make Yango sync run as a scheduled 04:00 AM job, not a manual 09:00 AM action — and the timing problem disappears.

Why this design

Targets stay simple (rides → tier → adjustment). Don't overengineer the math.
Flags surface specific actions. A Tier B rider with 🚩 Low Quality flag tells the Team Leader exactly what coaching they need — work on acceptance rate, not just push more rides.
Two riders can be in the same tier for very different reasons. A rider with 22 rides at 80% acceptance is in a different place than a rider with 22 rides at 35% acceptance. The flags expose this.

Where this lives in EFMS
Two new things to build on top of the dynamic target system:

Rider Performance Profile — a per-rider page showing all 5 dimensions with 30-day trend lines, current flags, and a "what to coach" auto-suggestion based on which flag is most impactful.
Performance Dashboard view — same Active Riders Dashboard we already designed, but with the flag icons added next to each rider name and filterable by flag. The Team Leader's Saturday tier-list meeting now has a clear "who needs what kind of help" view.

Both run off the same data the target system already uses — no new data sources required. Just new views on top.
